# TCP/UDP

## Q. TCP와 UDP의 차이점은 무엇인가요?

### 답변

TCP는 연결 지향 프로토콜로, 데이터를 전송하기 전에 3-way handshake를 통해 송신자와 수신자 간 연결을 수립합니다.

연결 이후에는 Sequence Number, ACK, 재전송, 흐름 제어, 혼잡 제어를 통해 데이터의 신뢰성과 순서를 보장합니다. 따라서 파일 전송, 웹 요청처럼 데이터가 정확하게 도착해야 하는 경우에 적합합니다.

반면 UDP는 비연결형 프로토콜로, 별도의 연결 수립 과정 없이 데이터를 바로 전송합니다. TCP처럼 데이터 도착 여부, 순서, 재전송을 기본적으로 보장하지 않기 때문에 신뢰성은 낮지만, 구조가 단순하고 지연 시간이 짧습니다.

따라서 신뢰성이 중요한 경우에는 TCP를 사용하고, 실시간성이 중요한 경우에는 UDP를 사용합니다.

### 비교

| 구분 | TCP | UDP |
| --- | --- | --- |
| 연결 방식 | 연결 지향 | 비연결형 |
| 신뢰성 | 보장 | 보장하지 않음 |
| 순서 보장 | 보장 | 보장하지 않음 |
| 재전송 | 지원 | 지원하지 않음 |
| 흐름 제어 | 지원 | 지원하지 않음 |
| 혼잡 제어 | 지원 | 지원하지 않음 |
| 속도 | 상대적으로 느림 | 상대적으로 빠름 |
| 사용 예시 | HTTP/HTTPS, FTP, SMTP | DNS, VoIP, 실시간 스트리밍, 온라인 게임 |


## Q. TCP를 사용하는 대표적인 예시는?

### 답변

대표적으로 HTTP, HTTPS, SMTP, FTP 등이 TCP를 사용합니다.

웹 서비스, 이메일, 파일 전송처럼 데이터가 중간에 손실되거나 순서가 바뀌면 문제가 되는 서비스에서는 신뢰성과 순서 보장이 중요하기 때문에 TCP를 사용합니다.

## Q. UDP를 사용하는 대표적인 예시는?

### 답변

대표적으로 DNS, VoIP, 실시간 스트리밍, 온라인 게임 등이 UDP를 사용합니다.

예를 들어 온라인 게임이나 음성 통화에서는 오래된 패킷을 재전송받는 것보다 최신 데이터를 빠르게 받는 것이 더 중요합니다. 그래서 일부 패킷 손실을 감수하더라도 지연 시간이 짧은 UDP를 사용합니다.

## Q. TCP가 신뢰성을 보장하는 방법은?

### 답변

TCP는 Sequence Number와 ACK를 이용해 데이터의 순서와 수신 여부를 확인합니다.

송신자는 각 데이터에 Sequence Number를 붙여 전송하고, 수신자는 정상적으로 받은 데이터에 대해 ACK를 응답합니다. 만약 일정 시간 안에 ACK가 오지 않거나 패킷 손실이 감지되면 TCP는 해당 데이터를 재전송합니다.

이를 통해 TCP는 데이터 손실을 복구하고, 순서가 뒤바뀐 패킷도 올바른 순서로 재조립할 수 있습니다.

## Q. TCP 3-way handshake는 무엇인가요?

### 답변

3-way handshake는 TCP 연결을 시작하기 전에 클라이언트와 서버가 서로 통신 가능한 상태인지 확인하고 초기 Sequence Number를 교환하는 과정입니다.

1. 클라이언트가 서버에 SYN 패킷을 보냅니다.
2. 서버는 SYN을 받고 SYN + ACK 패킷을 응답합니다.
3. 클라이언트가 ACK 패킷을 보내면 연결이 수립됩니다.

이 과정을 통해 양쪽 모두 데이터를 주고받을 준비가 되었음을 확인합니다.

## Q. TCP의 흐름 제어와 혼잡 제어는 무엇인가요?

### 답변

흐름 제어는 송신자가 수신자의 처리 능력보다 많은 데이터를 보내지 않도록 조절하는 기능입니다. 수신자는 자신이 받을 수 있는 버퍼 크기를 알려주고, 송신자는 그 범위 안에서 데이터를 전송합니다.

혼잡 제어는 네트워크 자체가 혼잡해지는 것을 방지하기 위한 기능입니다. 네트워크 상태에 따라 송신 속도를 조절해 패킷 손실과 지연 증가를 줄입니다.

즉, 흐름 제어는 수신자 보호가 목적이고, 혼잡 제어는 네트워크 보호가 목적입니다.

## Q. UDP는 항상 신뢰성이 없나요?

### 답변

UDP 자체는 신뢰성, 순서 보장, 재전송을 제공하지 않습니다.

하지만 애플리케이션 레벨에서 ACK, 재전송, 순서 정렬 같은 로직을 직접 구현하면 필요한 범위 안에서 신뢰성을 보완할 수 있습니다. 예를 들어 QUIC은 UDP 위에서 동작하지만 연결 관리, 암호화, 신뢰성 있는 전송 등을 자체적으로 제공합니다.

따라서 UDP는 신뢰성을 전혀 구현할 수 없는 프로토콜이라기보다, 기본 기능으로 제공하지 않는 프로토콜이라고 보는 것이 정확합니다.

## Q. TCP와 UDP 중 어떤 기준으로 선택하나요?

### 답변

데이터의 정확성과 순서가 중요하면 TCP를 선택합니다. 예를 들어 웹 페이지, 로그인 요청, 파일 다운로드, 이메일 전송은 데이터가 손실되면 안 되기 때문에 TCP가 적합합니다.

반대로 실시간성이 중요하고 일부 데이터 손실을 허용할 수 있다면 UDP를 선택합니다. 예를 들어 음성 통화, 영상 스트리밍, 온라인 게임에서는 오래된 데이터를 정확히 받는 것보다 최신 데이터를 빠르게 받는 것이 더 중요합니다.
