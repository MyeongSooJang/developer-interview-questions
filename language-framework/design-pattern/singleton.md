# Singleton

## Q. 싱글톤 패턴이란 무엇인가요?

### 답변

싱글톤 패턴은 특정 클래스의 인스턴스를 단 하나만 생성하도록 보장하고, 어디서든 그 인스턴스에 접근할 수 있게 해주는 디자인 패턴입니다.

구현 방식은 먼저 생성자를 private으로 선언해 외부에서 new로 객체를 만들 수 없게 막고, 클래스 내부에 자기 자신의 타입을 가지는 static 필드를 두어 인스턴스를 저장합니다. 그리고 getInstance() 같은 static 메서드를 통해 항상 동일한 인스턴스를 반환하도록 합니다.

주로 객체 생성 비용이 크거나, 여러 곳에서 공통된 자원을 공유해야 할 때 사용하며, 대표적인 예로는 DB 커넥션 풀이나 설정 정보를 관리하는 객체가 있습니다.

단점도 있습니다. 먼저 클라이언트가 getInstance()를 호출하는 과정에서 구체 클래스 이름을 직접 알아야 하기 때문에 결합도가 높아질 수 있습니다. 또한 private 생성자 때문에 테스트 시 Mock 객체로 대체하기가 어려워 단위 테스트 작성에 제약이 생길 수 있습니다.

그리고 인스턴스를 하나만 두고 공유하는 구조이기 때문에 stateful하게 설계하면 동시성 문제가 발생할 수 있습니다. 여러 스레드가 같은 인스턴스의 필드에 접근하면서 값을 덮어쓰거나 잘못 읽는 상황이 생길 수 있기 때문입니다.

따라서 싱글톤은 반드시 stateless하게 설계해야 합니다. 특정 클라이언트에 의존적인 필드나 값이 변경되는 필드를 두지 않아야 하며, 공유가 필요한 값은 인스턴스 필드 대신 지역변수나 파라미터, 또는 ThreadLocal을 활용하는 것이 안전합니다.

그리고 결합도가 높아지는 문제는 의존성 주입을 통해 해결할 수 있습니다. 대표적으로 스프링 컨테이너가 Bean을 싱글톤으로 관리하면서 필요한 곳에 주입해주는 방식이 있는데, 이를 통해 싱글톤의 장점은 유지하면서 결합도 문제를 함께 해결할 수 있습니다.
